Context
- What is the problem?
        - Customer tells you what they want
        - You get together a team
        - You get together a technology
        - Use that team to make the thing that the customer wants
        - Anything that can go wrong, will go wrong
        - Change
- Bricks without straw
- Insistence on a function list
- Arguments about "minimum viable product"
- Hard deadlines/Ambitious deadlines

Concept
- What is a value stream?
- How does sofware work in a value stream?
- How does getting funding for software work as a value stream?
        - This is often opaque
	- However it works it's different from a *delivery* value stream

Concrete Practice
- create pull
- pave the way
- baby steps
- be open to other value
- manage the emotions
- manage status
- understand the development value landscape - clients, funders

Conclusion
- it's very hard to make the transition from funded value stream to developed value stream smooth
- good project management is about managing this transition and making the developed value stream as valuable as possible *OR* making it obvious that the developed value stream
